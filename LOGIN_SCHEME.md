# Схема логинизации пользователя (Sphere)

Документ описывает варианты и рекомендуемую схему входа пользователя с учётом локальности и конфиденциальности приложения.

---

## Текущее состояние

- В БД есть таблица `users` (id, name, settings, created_at), но она **не используется**.
- Заметки, задачи, календарь, чат, база знаний **не привязаны к user_id** — по сути один общий «профиль» на установку.
- Конфиг (`config.yaml`) один на приложение, не per-user.

---

## Цели логинизации

1. **Несколько людей за одним компьютером** — каждый видит свои заметки, задачи, чаты.
2. **Защита доступа** — пароль или PIN при входе, чтобы посторонний не открыл данные.
3. **Опционально:** синхронизация между устройствами (позже, через свой бэкенд или E2E).

Для Sphere с акцентом на локальность разумно начать с **1 и 2** без облачного логина.

---

## Варианты

### A. Только локальные профили (без пароля)

- При первом запуске: «Введите имя» → создаётся запись в `users`, сохраняется `current_user_id` в конфиге/БД.
- Экран выбора профиля при старте: список пользователей, «Добавить профиль», «Войти».
- Все сущности получают `user_id`; запросы фильтруются по текущему пользователю.
- **Плюсы:** просто, без криптографии. **Минусы:** любой, кто запустит приложение, может выбрать любой профиль.

### B. Локальные профили + мастер-пароль на приложение

- Один общий пароль (или PIN) на всё приложение. При запуске — экран ввода пароля/PIN.
- Хранить только хэш (например bcrypt/argon2) в конфиге или в таблице `app_lock` (key=`pin_hash`, value=хэш).
- После ввода пароля — показывать выбор профиля (как в A) или сразу главный экран, если профиль один.
- **Плюсы:** защита от случайного доступа. **Минусы:** забытый пароль = сброс через удаление конфига/поля (нужна явная инструкция).

### C. Пароль на каждого пользователя

- В таблице `users` добавить `password_hash`, при входе в профиль — ввод пароля.
- **Плюсы:** разная защита у разных профилей. **Минусы:** сложнее UX (несколько паролей), больше кода.

### D. Облачный логин (OAuth / email+пароль)

- Вход через внешний сервис, синхронизация с облаком. Противоречит текущей философии «всё локально», оставляем на потом.

---

## Рекомендуемая схема (поэтапно)

### Фаза 1: Профили без пароля

1. **БД**
   - Во все таблицы с данными добавить `user_id INTEGER REFERENCES users(id)` (notes, tasks, calendar_events, knowledge_documents, chat_history, search_history).
   - В `users` оставить: id, name, settings, created_at. Опционально: avatar_path (локальный путь к картинке).

2. **Конфиг / состояние**
   - В конфиге или в таблице `settings`: ключ `current_user_id` (текущий пользователь).
   - При первом запуске, если пользователей нет: показать экран «Создать профиль» (имя), создать пользователя, записать `current_user_id`.

3. **Экраны**
   - **Старт приложения:** если `current_user_id` не задан или пользователь удалён — показать экран выбора/создания профиля (список + «Добавить»).
   - После выбора/создания — сохранить `current_user_id`, загрузить главный UI (sidebar + контент).

4. **Логика**
   - Все вызовы `get_notes`, `get_tasks`, `create_note`, и т.д. принимают/подставляют `user_id` текущего пользователя. Миграция: у существующих строк проставить `user_id = 1` (или создать дефолтного пользователя и привязать к нему старые данные).

### Фаза 2: Защита паролем/PIN (опционально)

1. В конфиг или таблицу `settings` добавить:
   - `app_lock_enabled: bool`
   - `app_pin_hash: str` (или `app_password_hash`) — хэш пароля/PIN для входа в приложение.

2. При запуске: если `app_lock_enabled` — показать экран ввода PIN/пароля; при успешной проверке хэша — переходить к выбору профиля или к главному экрану.

3. В настройках (или в «Личном кабинете»): пункт «Защита приложения» — включить/выключить PIN, смена PIN (старый → новый, сохранять новый хэш).

4. Библиотеки: `hashlib` + соль для простого PIN; для пароля лучше `passlib` (bcrypt) или `argon2-cffi`.

---

## Что править в коде (Фаза 1)

| Компонент | Изменения |
|-----------|-----------|
| `database.py` | Миграция: добавить `user_id` в таблицы; во все запросы (SELECT/INSERT/UPDATE) добавить условие/значение `user_id`. Добавить методы `get_users()`, `create_user(name)`, `get_user(id)`. |
| `config.py` | Поле `current_user_id: Optional[int] = None` (или хранить в `settings` в БД). |
| `app.py` | При старте проверять `current_user_id`; если нет — открывать экран выбора/создания профиля вместо главного. Передавать `user_id` в модули и в Database. |
| Новый UI | Экран «Выбор профиля» (список, кнопка «Добавить профиль»), экран «Новый профиль» (поле имени, «Создать»). |
| Модули (заметки, задачи, чат и т.д.) | Получать `user_id` из app/контекста и передавать в Database. |

---

## Резюме

- **Сейчас:** один общий «пользователь», таблица `users` не задействована.
- **Цель:** несколько локальных профилей на одну установку, опционально защита паролем/PIN.
- **Шаг 1:** ввести `user_id` в данные и экран выбора/создания профиля при старте.
- **Шаг 2:** по желанию — мастер-пароль/PIN на приложение с хранением хэша и экраном ввода при запуске.

Если нужно, следующий шаг — расписать миграцию SQL и сигнатуры методов в `database.py` под Фазу 1.
